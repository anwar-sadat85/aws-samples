# k6 Load Testing

This document covers the load testing setup for the Todo & Tasks app — the two k6 test
scripts, the shared library, the four load profiles, and the GitHub Actions workflow that
runs them automatically after every UAT deployment.

---

## How it is triggered

```
Amplify Build (success)
        │
        ▼
  Amazon EventBridge
  (rule: BUILD_SUCCESS on your Amplify app)
        │
        ▼
  Lambda — load-test-dispatcher
  (calls GitHub repository_dispatch API with event type: amplify-uat-deployed-todo)
        │
        ▼
  GitHub Actions — Load Tests — UAT
  (.github/workflows/load-test.yml)
        │
        ├─► Job 1: Smoke Tests  (always runs)
        │         ├── GraphQL smoke  ─┐
        │         └── REST smoke     ─┴─ parallel background processes
        │
        └─► Job 2: Load Tests  (runs only if smoke passes)
                  ├── GraphQL load   ─┐
                  └── REST load      ─┴─ parallel background processes
                        │
                        ├── Write job summary to GitHub Actions
                        ├── Upload reports/ artifact
                        └── Upload JSON reports to S3
```

The workflow can also be triggered manually from the GitHub Actions UI with a choice of
profile (`smoke`, `load`, `stress`, or `soak`). When triggered manually with `smoke`,
only Job 1 runs — Job 2 is skipped.

Push events to any path inside `amplify-todo-app/**` also trigger the workflow (always
runs the smoke profile in Job 1; Job 2 is skipped for push events).

## What gets tested

| Test | Target | Auth header | Operations |
|------|--------|-------------|------------|
| `graphql.test.js` | AppSync GraphQL | Raw ID token (no `Bearer`) | `listTodos`, `getTodo`, `createTodo`, `updateTodo`, `deleteTodo` |
| `rest.test.js` | API Gateway `/tasks` | `Bearer <id_token>` | `GET /tasks`, `POST /tasks`, `GET /tasks/{taskId}`, `DELETE /tasks/{taskId}` |

> **Why different auth header formats?**
> AppSync's Cognito USER_POOLS authorizer expects the raw ID token directly in the
> `Authorization` header. API Gateway's Cognito authorizer requires the standard
> `Authorization: Bearer <token>` format.

> **No PUT / PATCH on REST:** The Lambda handler (`amplify/functions/tasks-api/handler.ts`)
> only implements four routes. No update route exists; the REST test covers all four.

## Profiles

Select a profile by passing `--env PROFILE=<name>` to k6, or via the `profile` input
on a manual workflow dispatch. Defaults to `smoke` if unset.

| Profile | VUs | Duration | Purpose |
|---------|-----|----------|---------|
| `smoke` | 1 | 1 min (30 s + 30 s) | Verify scripts and endpoints are reachable. Always runs first in CI. |
| `load` | 0 → 20 → 0 | 7 min (1 m ramp + 5 m sustain + 1 m ramp-down) | Typical expected traffic. Validates steady-state performance. |
| `stress` | 0 → 20 → 50 → 100 → 150 → 0 | 8 min | Find the breaking point by incrementally increasing VUs beyond normal load. |
| `soak` | 0 → 10 → 0 | 30 min (2 m ramp + 26 m sustain + 2 m ramp-down) | Detect memory leaks, connection pool exhaustion, and gradual degradation. |

### Thresholds

Thresholds are enforced by k6. Exceeding any threshold causes the run to exit non-zero
and fail the CI step.

| Metric | GraphQL | REST | Error rate (smoke) | Error rate (load / soak) | Error rate (stress) |
|--------|---------|------|--------------------|--------------------------|---------------------|
| p95 latency | < 2 000 ms | < 1 500 ms | — | — | — |
| `graphql_errors` / `rest_errors` | — | — | < 5% | < 1% | < 10% |

The error metrics (`graphql_errors`, `rest_errors`) are custom `Rate` metrics defined in
`lib/checks.js`. They are used instead of k6's built-in `http_req_failed` because the
built-in metric tracks transport-level failures only and would miss application-level
errors returned with HTTP 200 (which AppSync always does).

## Operations covered

### GraphQL (`graphql.test.js`)

Each VU iteration executes five operations in sequence, with 0.5 s sleeps between them:

| # | Operation | GraphQL type | Key input fields |
|---|-----------|--------------|-----------------|
| 1 | `listTodos` | Query | — |
| 2 | `getTodo` | Query | `id` from `TEST_TODO_ID` env var |
| 3 | `createTodo` | Mutation | `title`, `description`, `completed: false` |
| 4 | `updateTodo` | Mutation | `id` (just-created), `title`, `completed: true` |
| 5 | `deleteTodo` | Mutation | `id` (just-created) |

Steps 4 and 5 are skipped for an iteration if `createTodo` fails. The `getTodo` step is
skipped if `TEST_TODO_ID` is not set.

All fields are derived from `amplify/data/resource.ts`. The Todo model fields used are:
`id`, `title`, `description`, `completed`, `createdAt`, `updatedAt`.

### REST (`rest.test.js`)

Each VU iteration executes four requests in sequence, with 0.5 s sleeps between them:

| # | Route | Method | Expected status | Body check |
|---|-------|--------|-----------------|------------|
| 1 | `/tasks` | GET | 200 | Non-empty |
| 2 | `/tasks` | POST | 201 | Non-empty — extracts `taskId` |
| 3 | `/tasks/{taskId}` | GET | 200 | Non-empty |
| 4 | `/tasks/{taskId}` | DELETE | 200 | Skipped |

Steps 3 and 4 are skipped for an iteration if `POST /tasks` fails to return a `taskId`.
DELETE returns HTTP 200 with `{"message": "Task deleted"}` — not 204.

## Shared library

### `lib/auth.js` — per-VU token cache

Calls Cognito `InitiateAuth` with `USER_PASSWORD_AUTH` and caches the ID token in a
module-level variable. Because each k6 VU runs its own JavaScript runtime, the variable
is effectively per-VU — Cognito is called at most once per VU regardless of how many
iterations the VU runs.

```
POST https://cognito-idp.{AWS_REGION}.amazonaws.com/
X-Amz-Target: AWSCognitoIdentityProviderService.InitiateAuth
Content-Type: application/x-amz-json-1.1

{
  "AuthFlow": "USER_PASSWORD_AUTH",
  "ClientId": "<COGNITO_CLIENT_ID>",
  "AuthParameters": { "USERNAME": "...", "PASSWORD": "..." }
}
```

Auth requests are tagged `cognito_initiate_auth` so they are excluded from the
`http_req_duration` percentile calculations in the test thresholds.

### `lib/checks.js` — check helpers and custom metrics

| Export | Type | Description |
|--------|------|-------------|
| `graphqlErrors` | `Rate` | Incremented when any GraphQL check fails |
| `restErrors` | `Rate` | Incremented when any REST check fails |
| `graphqlReqDuration` | `Trend` | Raw HTTP duration for every GraphQL request (ms) |
| `restReqDuration` | `Trend` | Raw HTTP duration for every REST request (ms) |
| `checkGraphQL(res, name)` | function | Asserts status 200, valid JSON, no `errors[]`, `data` present |
| `checkREST(res, status, checkBody)` | function | Asserts expected status; optionally asserts non-empty body |

### `lib/profiles.js` — stages and thresholds

Exports a single function:

```js
buildOptions(profileName, p95Ms, errorMetric)
// e.g.
buildOptions('load', 2000, 'graphql_errors')
buildOptions('load', 1500, 'rest_errors')
```

Returns a k6 `options` object with `stages` and `thresholds`. The `errorMetric`
parameter specifies which custom `Rate` metric to threshold on; it falls back to
`http_req_failed` if omitted.

## GitHub Actions workflow

**File:** `.github/workflows/load-test.yml`

### Triggers

| Event | Behaviour |
|-------|-----------|
| `repository_dispatch: amplify-uat-deployed-todo` | Runs smoke → load in sequence |
| `push` to `amplify-todo-app/**` | Runs smoke only (Job 2 skipped) |
| `workflow_dispatch` with `profile: smoke` | Runs smoke only (Job 2 skipped) |
| `workflow_dispatch` with `profile: load \| stress \| soak` | Runs smoke → selected profile |

### Concurrency

The workflow sets `concurrency.group: load-tests-uat-todo` with `cancel-in-progress: false`.
If a new deployment completes while a load test is already running, the dispatch is queued
rather than cancelling the in-progress run so results are never silently dropped.

### k6 installation

k6 is installed from the official GitHub release tarball — no third-party Actions are used:

```bash
curl -sL https://github.com/grafana/k6/releases/download/v0.54.0/k6-v0.54.0-linux-amd64.tar.gz \
  | tar -xz --strip-components=1 -C /usr/local/bin k6-v0.54.0-linux-amd64/k6
```

### Job summary

After the load-test job completes (even on failure), a markdown summary is written to
`$GITHUB_STEP_SUMMARY` using `jq` to parse the `--summary-export` JSON files. The
summary shows p95 latency, error rate, total request count, and threshold pass/fail for
each test.

### Artifacts and S3

| Output | Name / path | Retention |
|--------|-------------|-----------|
| Smoke reports artifact | `k6-smoke-reports-{run_id}` | 30 days |
| Load reports artifact | `k6-results-{run_id}` | 90 days |
| S3 key prefix | `amplify-todo-app/uat/{YYYY-MM-DD}/{run_id}/` | Per S3 bucket policy |

Two files are uploaded to S3: `graphql-load.json` and `rest-load.json`.

## Prerequisites

### 1 — GitHub secrets

Configure these in **Settings → Secrets and variables → Actions**.

| Secret | Description |
|--------|-------------|
| `COGNITO_CLIENT_ID` | Cognito app client ID (`amplify_outputs.json → auth.user_pool_client_id`) |
| `COGNITO_USER_POOL_ID` | Cognito user pool ID (`amplify_outputs.json → auth.user_pool_id`) |
| `COGNITO_TEST_USER` | Email of the dedicated load test account |
| `COGNITO_TEST_PASSWORD` | Password of the dedicated load test account |
| `APPSYNC_ENDPOINT` | AppSync GraphQL URL (`amplify_outputs.json → data.url`) |
| `APIGW_BASE_URL` | API Gateway base URL — no trailing slash |
| `TEST_TODO_ID` | UUID of a pre-seeded Todo record used for `getTodo` |
| `RESULTS_S3_BUCKET` | S3 bucket name for test result storage |
| `AWS_ROLE_ARN` | IAM role ARN assumed via OIDC |
| `AWS_REGION` | AWS region (e.g. `ap-southeast-2`) |

### 2 — Dedicated load test account

Create a permanent Cognito user used only for load testing. Never use a real user's
credentials. The same `USER_PASSWORD_AUTH` flow is used as in the DAST workflow.

```bash
aws cognito-idp admin-create-user \
  --user-pool-id <USER_POOL_ID> \
  --username load-test@example.com \
  --temporary-password "TempP@ss1" \
  --message-action SUPPRESS \
  --region ap-southeast-2

aws cognito-idp admin-set-user-password \
  --user-pool-id <USER_POOL_ID> \
  --username load-test@example.com \
  --password "Perm@nentP@ss1!" \
  --permanent \
  --region ap-southeast-2
```

Store the email as `COGNITO_TEST_USER` and the permanent password as
`COGNITO_TEST_PASSWORD` in GitHub Secrets.

### 3 — Pre-seeded test record (`TEST_TODO_ID`)

The `getTodo` query requires a Todo that already exists in the target environment.
Create one using the app UI or the AppSync console after a UAT deployment and store
its `id` as `TEST_TODO_ID`. The record is read-only in the test — it is never
modified or deleted.

### 4 — GitHub Actions IAM role (OIDC)

The load-test job assumes an IAM role via OIDC to upload results to S3.
The role needs `s3:PutObject` on the results bucket and `cognito-idp:InitiateAuth`
on the user pool (the same requirement as DAST — a single role can serve both workflows).

**Trust policy:** same as the DAST role — see [DAST.md](DAST.md#3--github-actions-iam-role-oidc).

**Permission policy:**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "cognito-idp:InitiateAuth",
      "Resource": "arn:aws:cognito-idp:ap-southeast-2:<ACCOUNT_ID>:userpool/<USER_POOL_ID>"
    },
    {
      "Effect": "Allow",
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::<RESULTS_S3_BUCKET>/amplify-todo-app/*"
    }
  ]
}
```

### 5 — `USER_PASSWORD_AUTH` on the user pool client

The same CDK escape hatch documented in [DAST.md](DAST.md#user-pool-client--user_password_auth)
enables this auth flow. No additional configuration is required if DAST is already set up.

## Running manually (workflow dispatch)

1. Go to **Actions → Load Tests — UAT** in the GitHub repository.
2. Click **Run workflow**.
3. Select the branch and choose a profile from the dropdown.
4. Click **Run workflow**.

Selecting `smoke` runs only Job 1 and exits. Selecting `load`, `stress`, or `soak`
runs the smoke gate first, then the selected profile.

## Running locally

### Smoke

```bash
k6 run \
  --env PROFILE=smoke \
  --env COGNITO_CLIENT_ID=<client-id> \
  --env COGNITO_TEST_USER=load-test@example.com \
  --env COGNITO_TEST_PASSWORD=<password> \
  --env APPSYNC_ENDPOINT=https://<id>.appsync-api.ap-southeast-2.amazonaws.com/graphql \
  --env TEST_TODO_ID=<uuid> \
  amplify-todo-app/tests/load/graphql.test.js
```

```bash
k6 run \
  --env PROFILE=smoke \
  --env COGNITO_CLIENT_ID=<client-id> \
  --env COGNITO_TEST_USER=load-test@example.com \
  --env COGNITO_TEST_PASSWORD=<password> \
  --env APIGW_BASE_URL=https://<id>.execute-api.ap-southeast-2.amazonaws.com/prod \
  amplify-todo-app/tests/load/rest.test.js
```

### With JSON report export

```bash
mkdir -p reports

k6 run \
  --env PROFILE=load \
  --env COGNITO_CLIENT_ID=<client-id> \
  --env COGNITO_TEST_USER=load-test@example.com \
  --env COGNITO_TEST_PASSWORD=<password> \
  --env APPSYNC_ENDPOINT=https://... \
  --env TEST_TODO_ID=<uuid> \
  --summary-export=reports/graphql-load.json \
  amplify-todo-app/tests/load/graphql.test.js
```

### Both tests in parallel (same terminal)

```bash
mkdir -p reports

k6 run --env PROFILE=load ... \
  --summary-export=reports/graphql-load.json \
  amplify-todo-app/tests/load/graphql.test.js &

k6 run --env PROFILE=load ... \
  --summary-export=reports/rest-load.json \
  amplify-todo-app/tests/load/rest.test.js &

wait
```
